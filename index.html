Manifesto

You do not need to know how to code. You need to know how to ask.

Vibecoding is not the act of writing syntax. It is the act of directing intelligence through language — describing what you want with enough clarity that the machine can meet you there. The keyboard becomes a place for intent, not for memorising brackets and semicolons.

This is a course for absolute beginners. Three chapters, no prerequisites, no prior line of code required. You will learn to speak to your tools, to choose them like fabric, and to release a first project into the world — not because it is finished, but because it is alive.

I
Chapter I — The Prompt

The Prompt is your voice.

A prompt is a brief, not a search query. You are not typing keywords into a box — you are commissioning a piece of work.

  1. Give context first, instructions second

    Set the scene before you make the request. The model reasons better when it knows what world it is working inside.

  2. Precision reads as confidence

    Vague asks return vague work. Name the format, the tone, the constraints. Specificity is a kindness to your tools.

  3. Iterate like an editor, not a critic

    Refine in passes. Redirect rather than reject. Each round moves the draft closer, the way an editor shapes a manuscript.

  4. Show, don't just tell

    Paste an example, a reference, a shape you admire. A single sample communicates more than a paragraph of description.

II
Chapter II — The Instruments

The Instruments — chosen like fabric.

Three instruments carry a project from thought to release. Each is chosen for the moment it serves.

Claude
The Conversation

Where the thinking happens and the first version is written. You describe, it drafts; together you shape the idea into something real.

Cursor
The Workshop

The editor for precise control of the code. When the conversation needs hands, this is the bench where you tailor every seam.

Netlify
The Runway

Deployment — the project's walk into the world. One step and the work leaves the studio to be seen from anywhere.

III
Chapter III — The First Project

The First Project — released, not finished.

Your first project is a rehearsal for every one after it. Keep it small, describe it fully, and let it go.

  1. Choose something small enough to finish this week

    Scope is the difference between a project and a dream. Pick something you can hold in one hand.

  2. Describe the whole thing before you build any of it

    Write the brief in full first. A clear description is half the build already done.

  3. Build in public view, even if the audience is only you

    Work where you can see the whole shape. Visibility keeps you honest about what is finished and what is not.

  4. Release before it feels ready

    Ready is a feeling that rarely arrives. Ship at the edge of discomfort — that is where the real project begins.

Studio Notes

Six notes from the floor.

01

Describe what the code should become — not what it should avoid. Aim at the target, not away from the miss.

02

A vague prompt is a draft, not a failure. Rewrite it before you judge the result it produced.

03

Save a working version before you try a risky one. A known-good copy makes courage cheap.

04

Write your prompt like an instruction to a trusted assistant, not like a search query fired into a box.

05

If the result is wrong, the task was phrased wrong — it is not that your hands are in the wrong place.

06

Keep an archive of prompts that worked. It is worth more than any library of memorised syntax.

Enroll

Enrollment is by instinct, not application.
